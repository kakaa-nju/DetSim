# brk() 与 mmap()



**`brk(2)` 用来调整单一“堆（heap）段”的末尾；而 `mmap(2)` 则可以在进程地址空间中的任意位置创建或删除独立的内存区域（VMA）。**
 两者最终都会创建内核中的 VMA（虚拟内存区域），但使用方式和约束完全不同。



##  program break

```
Low addresses ──►                         High addresses
┌──────────────┬──────────┬────────────┬──────────┬──────────┐
|  text / ro   |  data    |  bss       |  heap    |  ...     |
| (file‑map)   | (file)   | (file)     |  <--brk  |          |
└──────────────┴──────────┴────────────┴──────────┴──────────┘
```

- 内核会创建一个 **从 `.bss` 结束之后开始，到当前 program break 为止** 的 VMA，整个堆区就是它。
- `brk(addr)` 的作用是：请求内核将这个 VMA 的“结尾地址”向上或向下移动：
  - 要求 `addr` ≥ 初始堆起始地址
  - 并且 `addr` 必须小于堆上方的第一个已有映射区域的起始地址。
- 如果 `addr` 已经跨越了一个已经被 `mmap` 映射占用的页，就会失败并返回 `ENOMEM`，也就是你提到的“跨 mmap 页面就失败”的原因。



## `mmap` 做了什么？

```text
... heap  ◄────brk
           │
           │  (gap that the kernel may fill)
           ▼
┌─────────────────────────────────────────────────────────────┐
│ new VMA: anonymous/file‑backed, fixed or anywhere           │◄─ mmap() creates this
└─────────────────────────────────────────────────────────────┘
```

- 每次成功的 `mmap()` 调用，都会创建一个**独立的 VMA**，它不会和堆共享地址空间。
- 除非你显式地传入 `MAP_FIXED`，否则内核会自动选择一个不会和已有内存区域冲突的地址。
- 因此，`mmap` 无法控制或推动 program break 的增长；但一旦 `mmap` 把映射放在堆上方，就会**阻止 `brk` 再向上扩展**。



## 哪些区域来自 `mmap`？哪些来自 `brk`？

| 区域（以 x86-64 上 ELF 程序为例）            | 创建方式                                 | 说明                          |
| -------------------------------------------- | ---------------------------------------- | ----------------------------- |
| **text / rodata / relro**                    | 可执行文件的 `mmap` 映射（只读、可执行） | 由动态加载器完成              |
| **.data / .bss（初始内容）**                 | 可执行文件的 `mmap` 映射（可写）         | 同样是文件映射                |
| **Heap（ `[heap]` in `/proc/<pid>/maps` ）** | 通过 `brk` 调整单个 VMA                  | glibc 的 `malloc` 默认使用它  |
| **Thread stacks**                            | 自动 `mmap`（匿名，可增长）              | 每个线程一个栈，并有保护页    |
| **Shared libraries**                         | `mmap` 加载 `.so` 文件                   | 由动态链接器负责              |
| **User `mmap` / `munmap`**                   | 显式调用                                 | 适合大块内存、JIT代码、文件等 |
| **vdso、vsyscall、ELF 辅助向量页等**         | 内核自动使用 `mmap`                      | 通常只读，微小                |
| **TLS 和动态加载器的私有堆空间**             | 混合机制                                 | 内部实现细节                  |



## 用户空间中的交互（以 glibc 的 `malloc` 为例）

| 分配大小                                                     | 默认策略             | 原因                             |
| ------------------------------------------------------------ | -------------------- | -------------------------------- |
| **≤ 128 KiB**                                                | 使用 `brk` 扩展堆    | 连续内存更紧凑，减少系统调用开销 |
| **> 128 KiB**                                                | 使用匿名 `mmap` 分配 | 大块内存可以用 `munmap` 快速释放 |
| Tweaked via `MALLOC_MMAP_THRESHOLD_`, `MALLOC_TRIM_THRESHOLD_`, `MALLOC_ARENA_MAX`, etc. |                      |                                  |

注意：Linux 对内存分配是**延迟和过度分配（overcommit）**的，这意味着程序可能分配了超出物理内存或 `brk` 限制的内存，但内核不会立即拒绝；而 `RLIMIT_AS` 限制的是所有 VMA 的总大小。



## 为什么 `brk` 增长会失败？

1. 已有 `mmap` 映射占住了堆的上方
2. 超出了 `RLIMIT_DATA` —— 只对 `brk` 管理的堆有效。
3. 安全机制限制 —— 如不能越过用户栈（尤其在 32 位平台）。
4. 虚拟地址空间耗尽 —— 在 64 位系统上罕见，在 32 位上常见（因为地址空间仅约 3 GiB）。

一旦 `brk` 失败，glibc 的 `malloc` 会自动切换到用 `mmap` 分配，这也是为什么程序通常不会因此崩溃。



## 编程时如何选择使用 `brk` 还是 `mmap`

| 需求                                                 | 推荐方式             |
| ---------------------------------------------------- | -------------------- |
| 想要连续的线性堆空间                                 | 使用 `brk`           |
| 想要多个独立区域（如大内存、文件映射、可保护的内存） | 使用 `mmap`          |
| 想要轻松释放、修改保护权限的内存区域                 | 使用 `mmap`          |
| 不关心分配细节，使用库如 `malloc` 即可               | 由其内部决定使用何者 |

一般来说，在现代应用中，几乎不需要手动调用 `brk`。应使用 `mmap` 或标准库，除非实现内存管理器或运行时系统。
